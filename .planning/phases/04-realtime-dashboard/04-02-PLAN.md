# 04-02: Session Broadcast Manager

## Objective

Create a room-based broadcast manager that maps WebSocket connections to session rooms, pushes attendance updates when students check in, and cleans up on disconnect.

## Tasks

### Task 1: Create broadcast manager

<read_first>
- server.ts (from 04-01)
- src/lib/ws-types.ts (from 04-01)
- src/lib/checkin-service.ts
</read_first>

<action>
Create `src/lib/ws-broadcast.ts`:

1. Export singleton `BroadcastManager` class:
   - `rooms: Map<string, Set<WebSocket>>` — sessionId → connected clients
   - `clientSessions: Map<WebSocket, Set<string>>` — reverse lookup for cleanup

2. Methods:
   - `handleConnection(ws: WebSocket)` — register new connection
   - `handleMessage(ws: WebSocket, raw: string)` — parse JSON, dispatch by type
   - `handleDisconnect(ws: WebSocket)` — remove from all rooms, clean up
   - `subscribe(ws: WebSocket, sessionId: string)` — add to room
   - `unsubscribe(ws: WebSocket, sessionId: string)` — remove from room
   - `broadcast(sessionId: string, message: WsMessage)` — send to all in room, handle send errors (remove dead clients)
   - `broadcastAttendanceUpdate(sessionId: string, checkedInCount: number, totalStudents: number, newCheckIn?: { studentId: string, studentName: string })` — convenience wrapper

3. Message dispatch (handleMessage):
   - `subscribe` → call `subscribe(ws, payload.sessionId)`
   - `unsubscribe` → call `unsubscribe(ws, payload.sessionId)`
   - `pong` → update ws's last-pong timestamp (for heartbeat tracking)
   - Unknown type → send `{ type: "session_error", payload: { message: "Unknown message type" } }`
</action>

<acceptance_criteria>
- `src/lib/ws-broadcast.ts` exists
- BroadcastManager is exported as a singleton
- `subscribe`/`unsubscribe` manage room membership correctly
- `broadcast` sends to all clients in a room (excluding dead ones)
- `handleDisconnect` cleans up all room entries for the disconnected client
- TypeScript: no type errors
</acceptance_criteria>

### Task 3: Wire broadcast into check-in submission

<read_first>
- src/lib/checkin-service.ts
- src/lib/ws-broadcast.ts (Task 1)
- src/app/api/checkin/submit/route.ts
</read_first>

<action>
1. In `submit/route.ts`, after successful `submitAttendance()`:

```typescript
// After attendance is recorded, broadcast update via WebSocket
// (non-blocking — don't fail the request if broadcast fails)
try {
  const { checkedInCount, totalStudents } = await getSessionStats(sessionId);
  broadcastManager.broadcastAttendanceUpdate(sessionId, checkedInCount, totalStudents, {
    studentId: student.id,
    studentName: student.name,
  });
} catch {
  // Broadcast errors are non-fatal; client falls back to polling
}
```

2. Import `BroadcastManager` in the route.

3. Add `getSessionStats()` helper to `checkin-service.ts`:
   - Query `SELECT count(*) FROM attendance_records WHERE session_id = $1`
   - Query `SELECT count(*) FROM students WHERE course_id = (SELECT course_id FROM check_in_sessions WHERE id = $1)`
   - Return `{ checkedInCount: number, totalStudents: number }`
</action>

<acceptance_criteria>
- `submit/route.ts` calls `BroadcastManager.broadcastAttendanceUpdate()` after successful check-in
- Broadcast is wrapped in try-catch — API response never fails due to WS broadcast

- TypeScript: no type errors
</acceptance_criteria>
