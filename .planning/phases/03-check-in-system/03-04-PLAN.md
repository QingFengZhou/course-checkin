<task type="auto">
  <name>Task 1: Create student check-in landing page (QR scan destination with form)</name>
  <files>src/app/checkin/page.tsx</files>
  <behavior>
    - Public page (no auth gate) at /checkin, reads session query parameter
    - On mount: reads `session` query param from URL (e.g., /checkin?session=abc123)
    - If no session param: show error "无效的签到链接"
    - Calls GET /api/checkin/[token] to resolve session
    - If session not found/expired: show "签到已结束" message
    - If session active: display form with course name, student ID input, name input, submit button
    - On submit: POST to /api/checkin/submit with { sessionToken, studentId, name }
    - If success: redirect to /checkin/success with student name in query params
    - If error (student not found): show error message inline
    - Mobile-optimized: large touch targets, single-column layout, no horizontal scroll
    - Loading states on form submission (disable button, show spinner)
  </behavior>
  <action>
    Create src/app/checkin/page.tsx as a Client Component ("use client") — reads URL params and makes API calls.

    This is a PUBLIC page — no auth middleware, no protected route group. Place directly under src/app/checkin/ (outside (protected) and (public) route groups).

    Layout (mobile-first, matching the login page style from Phase 1):
    - White card centered on gray background
    - Course name in large bold text at top
    - Two input fields stacked vertically: "学号" (type="text"), "姓名" (type="text")
    - Full-width submit button (blue, ~48px height for touch target)
    - Error message in red below form if submission fails

    Session resolution flow:
    1. On mount, read `session` from useSearchParams()
    2. If no session param: show error state "无效的签到链接"
    3. Fetch GET /api/checkin/[session]
    4. If 404: show "签到已结束" — dead end for expired sessions
    5. If 200: render form with courseName from response

    Submission flow:
    1. Validate: studentId and name are non-empty
    2. POST /api/checkin/submit with { sessionToken, studentId, name }
    3. If response.ok with alreadyCheckedIn: show "您已经签到过了" message
    4. If response.ok with studentName: redirect to /checkin/success?name={studentName}&course={courseName}
    5. If 400 with "Student not found": show "未找到该学生，请检查学号和姓名"
    6. If 400 with "Session expired": show "签到已结束"
    7. If 500: show "提交失败，请重试"

    Use useSearchParams from next/navigation to read the `session` query parameter. Wrap in Suspense boundary (Next.js 16 requirement for useSearchParams in client components).

    Follow the login page form pattern from src/app/(public)/login/page.tsx for input styling and layout consistency.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Student browser -> /checkin page | Public — no auth, session token in URL query param |
| Student browser -> /api/checkin/submit | Public — student identity verified by studentId+name match |
| Student browser -> /checkin/success | Public — no sensitive data, only confirmation display |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-13 | Spoofing | /api/checkin/submit | mitigate | Student must provide BOTH studentId AND name matching a course record — guessing one field is insufficient |
| T-03-14 | Information Disclosure | Session token in URL | mitigate | 5-minute expiry (D-3.01) limits window; token is nanoid-generated, not guessable |
| T-03-15 | Repudiation | Student check-in | mitigate | Attendance record links to student DB record with checked_at timestamp — auditable |
| T-03-SC | Tampering | npm/pip/cargo installs | mitigate | slopcheck + blocking human checkpoint for [ASSUMED]/[SUS] |
</threat_model>
