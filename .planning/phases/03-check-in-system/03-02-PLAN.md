  <files>src/app/api/sessions/route.ts, src/lib/checkin-types.ts</files>
  <behavior>
    - POST handler: creates new check-in session for a course
    - Auth gate: getAuthSession(request), return 401 if not authenticated
    - Validate request body with createSessionSchema (courseId as UUID)
    - Ownership check: verify course exists AND course.teacherId === session.user.id
    - Call createSession(courseId, teacherId) from checkin-service
    - Return { data: { token, sessionId, expiresAt } } on success
    - Follow Phase 2 route patterns: try/catch, NextResponse.json, Zod .issues API
  </behavior>
  <action>
    Create src/app/api/sessions/route.ts.

    Import pattern (match Phase 2 routes):
    ```
    import { NextRequest, NextResponse } from "next/server";
    import { db } from "@/db";
    import { courses } from "@/db/schema";
    import { eq } from "drizzle-orm";
    import { getAuthSession } from "@/lib/auth";
    import { createSessionSchema } from "@/lib/zod-schemas";
    import { createSession } from "@/lib/checkin-service";
    ```

    POST handler logic:
    1. Call getAuthSession(request) — return 401 if !isAuthenticated or !user
    2. Parse request body with createSessionSchema.safeParse() — return 400 if invalid

    4. If course not found — return 403 with "Course not found or access denied"
    5. Call createSession(courseId, session.user.id) — returns session with token
    6. Return 201 with { data: { token: session.token, sessionId: session.id, expiresAt: session.expiresAt } }
    7. Wrap in try/catch — return 500 on unexpected errors

    Also create/extend src/lib/checkin-types.ts with an ApiSessionResponse type for the API response shape.

<task type="auto">
  <name>Task 2: Create POST /api/checkin/submit route for student check-in (public, idempotent)</name>
  <files>src/app/api/checkin/submit/route.ts</files>
  <behavior>
    - POST handler: student submits attendance (no auth required)
    - Validate request body with submitAttendanceSchema (sessionToken, studentId, name)
    - Call getActiveSession(sessionToken) — return 404 if session not found/expired
    - Look up student by studentId + name within the session's course (via students table JOIN)
    - If student not found in course — return 400 with "Student not found in this course"
    - Call submitAttendance(session.id, student.dbId) — insert attendance_record
    - If unique constraint violation (duplicate) — return 200 with { data: { alreadyCheckedIn: true } } (idempotent, not an error)
    - On success — return 200 with { data: { studentName, checkedAt } }
    - No console.log (TypeScript coding-style rule)
  </behavior>
  <action>
    Create src/app/api/checkin/submit/route.ts. This is a PUBLIC endpoint — no auth gate.

    Import pattern:
    ```
    import { NextRequest, NextResponse } from "next/server";
    import { db } from "@/db";
    import { students, checkInSessions } from "@/db/schema";
    import { eq, and } from "drizzle-orm";
    import { submitAttendanceSchema } from "@/lib/zod-schemas";
    import { getActiveSession, submitAttendance } from "@/lib/checkin-service";
    ```

    POST handler logic:
    1. Parse request body with submitAttendanceSchema.safeParse() — return 400 if invalid
    2. Call getActiveSession(parsed.data.sessionToken) — if null, return 400 with "Session expired or not found" (CHECKIN-04)
    3. Look up student: select from students where studentId = parsed.data.studentId AND name = parsed.data.name AND courseId = session.courseId
    4. If no student found — return 400 with "Student not found in this course. Please check your student ID and name."
    5. Call submitAttendance(session.id, student.id)
    6. If returns { error: 'already checked in' } — return 200 with { data: { alreadyCheckedIn: true, message: "You have already checked in" } } (idempotency per CONTEXT.md Claude's Discretion)
    7. On success — return 200 with { data: { studentName: student.name, checkedAt: new Date().toISOString() } }
    8. Wrap in try/catch — return 500 on unexpected errors

    Student lookup must match BOTH studentId AND name to prevent guessing. This matches the project constraint: "扫码 + 学号/姓名确认身份 — 简单可靠".
  </action>
  <verify>
    <automated>npx tsc --noEmit --pretty 2>&1 | head -20</automated>

      3. POST /api/checkin/submit with wrong studentId/name returns 400 "Student not found"
      4. POST /api/checkin/submit with valid data returns 200 with studentName
      5. POST /api/checkin/submit again with same data returns 200 with alreadyCheckedIn: true

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Student browser -> /api/checkin/submit | Untrusted input without auth — must validate student belongs to course |
| Teacher browser -> /api/sessions | Authenticated — must verify course ownership before creating session |
| Session token in URL | Visible in browser history, logs — but 5-min expiry limits exposure window (D-3.01) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Authorization Bypass | POST /api/sessions | mitigate | Ownership check: query courses WHERE id=courseId AND teacherId=session.user.id before calling createSession |
| T-03-07 | Spoofing | POST /api/checkin/submit | mitigate | Student must provide BOTH studentId AND name matching a record in the session's course — prevents guessing with just an ID |

| T-03-04 | Denial of Service | submitAttendance API | accept | Same as Plan 01 — PostgreSQL MVCC handles 300 concurrent writes |
