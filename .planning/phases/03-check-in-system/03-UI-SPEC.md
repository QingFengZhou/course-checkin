## Interaction Patterns

### 1. Teacher Starts Check-In

```
Dashboard: Course Card
  └─ Click "发起签到"
      ├─ POST /api/sessions { courseId }
      ├─ Server creates session row + returns { token }
      └─ Client navigates to /checkin/[courseId]?session=<token>
          ├─ TeacherCheckInPage renders:
          │   ├─ QRCodeDisplay (QRCodeSVG with URL /checkin?session=<token>)
          │   ├─ SessionTimer (countdown from expiresAt)
          │   ├─ CheckInStatusBar ("已签到 0 / 30 人")
          │   └─ "结束签到" button (destructive, with confirm dialog)
          └─ Student opens URL via phone camera scan
```

**States:**
- **Loading:** Skeleton spinner while session is being created
- **Active:** QR code visible, timer counting down, status bar updating
- **Expired:** Timer reaches 0, QR code removed, "签到已结束" message, "返回仪表盘" link
- **Teacher-ended:** QR code removed, countdown stopped, "签到已结束" message
- **Error (session creation failed):** Error banner "创建签到失败，请重试" + "返回仪表盘" link
- **Error (polling fails):** Status bar shows "加载失败" with retry button (do NOT take down the page)

### 2. Student Checks In via QR Code

```
Phone camera scans QR code
  └─ Opens /checkin?session=<token>
      ├─ Fetches session info (course name) from GET /api/checkin/[token]
      ├─ Renders StudentCheckInLanding:
      │   ├─ Shows course name
      │   ├─ Student ID input
      │   ├─ Name input
      │   └─ "签到" submit button
      ├─ POST /api/checkin/submit { sessionToken, studentId, name }
      │   ├─ Success (201): Navigate to /checkin/success
      │   ├─ Already checked in (409): Show "你已签到，无需重复提交"
      │   ├─ Invalid session/expired (400/410): Show "签到已结束或无效"
      │   └─ Student not found in course (404): Show "未找到该学生，请确认学号和姓名"
      └─ CheckInSuccessPage:
          ├─ Checkmark icon
          ├─ "签到成功！" heading
          ├─ Student name + course name
          └─ "页面可关闭" instruction
```

**States:**
- **Loading (landing):** Spinner while fetching session info
- **Ready (landing):** Form visible, inputs focused
- **Submitting:** Button shows "提交中..." with disabled state and spinner animation
- **Success:** Full-page success view with checkmark — no auto-redirect (student keeps visual confirmation)
- **Error (session expired):** Expired session message — "签到已结束，请联系老师"
- **Error (session not found):** "无效的签到链接"
- **Error (network):** "网络错误，请重试" with retry button
- **Error (already checked in):** Non-blocking message "你已签到", with student name shown
- **Error (student not found):** "学号或姓名不匹配，请确认后重试" with retry form (clear inputs except student ID)

### 3. Teacher Manually Marks Student Present

```
Dashboard: StudentRoster modal (with active session)
  └─ Per student row: "签到" button
      ├─ POST /api/sessions/[sessionId]/manual-checkin
      │   Body: { studentId }
      ├─ Success (201): Button changes to checkmark + "已签到" (non-interactive)
      └─ Error (session expired): Show toast "签到已结束"
```

**States:**
- **No active session:** "签到" buttons hidden or disabled with tooltip "当前无进行中的签到"
- **Active session, not yet checked in:** Blue "签到" button, clickable
- **Already checked in:** Green checkmark + "已签到" text (non-interactive)
- **Manual check-in fails:** Inline error on the row (red text, fade out after 3 seconds)

### 4. Session Timeout

- **Expiry check:** On every API call, server checks `expires_at < now()`. No background job.
- **Teacher side:** SessionTimer component decrements in 1s intervals. When expiresAt reached, local state transitions to "expired" — QR removed, status changes to "签到已结束".
- **Student side:** On form submission attempt, server returns 410 Gone if session expired. Student sees "签到已结束".
- **Edge case (page refresh):** If teacher refreshes expired check-in page, the server-side check returns expired status. Page renders expired state.

### 5. Navigation

- **Teacher:** `/checkin/[courseId]` — self-auth required (cookies() + verifyToken), redirect to `/login` if unauthenticated
- **Student (public):** `/checkin?session=<token>` — no auth, public
- **Success (public):** `/checkin/success` — no auth, public
- **Post-check-in flow:** No redirect — student keeps success confirmation open
- **Back navigation:** Teacher check-in page has a "返回仪表盘" link in page header

## Open Questions

1. **Student list on teacher check-in page:** Should the check-in page embed a full student status list (as shown in the layout above), or should it only show the QR code + count, with manual check-in done via the Dashboard roster? Recommendation: embed a simplified status list in the check-in page for teacher convenience — avoids route switching. This is documented in the layout above but needs planner/executor confirmation.

2. **No polling API for attendance count:** RESEARCH.md identified that no API endpoint exists for the teacher page to fetch "X checked in / Y total." Three options:
   - (a) Extend public GET /api/checkin/[token] to include attendance stats when auth cookie present
   - (b) Add teacher-only GET /api/sessions/[sessionId] endpoint
   - (c) Add `?includeCount=true` param to the existing public endpoint
   - **Recommendation:** Option (b) — clean separation of concerns, follows existing Phase 2 pattern of teacher-only endpoints.
